# King of Worm - 프로젝트 분석

## 1. 프로젝트 개요

**이름:** King of Worm
**버전:** 1.0.0
**타입:** 실시간 멀티플레이어 웹 게임
**게임 컨셉:** 뱀(웜)을 조종하여 먹이를 먹고 성장하며, 다른 뱀의 몸에 부딪히면 죽고, 살아남아 1위를 차지하는 게임

## 2. 기술 스택

| 구분 | 기술 |
|------|------|
| 서버 런타임 | Node.js 20 (Alpine) |
| 웹 프레임워크 | Express 4.18 |
| 실시간 통신 | WebSocket (`ws` 8.16) |
| 렌더링 | HTML5 Canvas 2D |
| 배포 | Docker / Docker Compose |
| 광고 | Google AdSense (`ca-pub-2601124884291683`) |

## 3. 프로젝트 구조

```
slither/

```

## 4. 아키텍처

### 4.1 두 가지 게임 모드

| 모드 | 파일 | 설명 |
|------|------|------|
| **멀티플레이** | `server.js` + `public/` | 서버에서 게임 로직 처리, 클라이언트는 렌더링만 |
| **로컬 싱글플레이** | `localGame.html` | 단일 HTML 파일에 게임 로직+렌더링 모두 포함 (서버 불필요) |

### 4.2 멀티플레이 아키텍처 (Authoritative Server)

```
[클라이언트 (game.js)]                [서버 (server.js)]
    |                                       |
    |-- join {name} ----------------------->|  WebSocket 연결
    |<-- welcome {id, worldSize, palettes} -|
    |                                       |
    |-- input {angle, boost} (50ms 간격) -->|  입력 전송
    |                                       |  게임 틱 처리 (30Hz)
    |<-- state {snakes, foods, lb, mm} -----|  상태 브로드캐스트 (15Hz)
    |                                       |
    |-- respawn {name} ------------------->|  리스폰
    |<-- death {score, length, kills} -----|  사망 알림
```

- **서버:** 모든 게임 로직 처리 (이동, 충돌, 먹이, AI)
- **클라이언트:** 입력 전송 + 렌더링 전용 (서버 상태를 받아 그림)

### 4.3 네트워크 최적화

- **컴팩트 JSON 포맷:** 키 이름을 1글자로 축약 (`t`, `s`, `f`, `y`, `lb`, `mm`)
- **세그먼트 샘플링:** 뱀의 세그먼트를 최대 60개로 다운샘플링하여 전송
- **좌표 정수화:** `(x + 0.5) | 0`으로 소수점 제거
- **뷰 범위 필터링:** 클라이언트별 VIEW_RANGE(1800px) 내 데이터만 전송
- **틱/전송 분리:** 게임 로직 30Hz, 네트워크 전송 15Hz

## 5. 게임 상수 (서버)

| 상수 | 값 | 설명 |
|------|----|------|
| `WORLD_SIZE` | 6000 | 월드 크기 (6000x6000 px) |
| `TICK_RATE` | 30 | 게임 루프 주기 (30 FPS) |
| `SEND_RATE` | 15 | 네트워크 전송 주기 (15 FPS) |
| `FOOD_COUNT` | 800 | 맵 내 먹이 수 |
| `AI_COUNT` | 15 | AI 뱀 수 (항상 유지) |
| `INITIAL_LENGTH` | 20 | 초기 뱀 길이 (세그먼트 수) |
| `BASE_SPEED` | 3.2 | 기본 이동 속도 |
| `BOOST_SPEED` | 6 | 부스트 이동 속도 |
| `TURN_RATE` | 0.12 | 회전 속도 (보간 계수) |
| `VIEW_RANGE` | 1800 | 클라이언트 가시 범위 |
| `GRID_CELL` | 200 | 공간 분할 그리드 셀 크기 |

## 6. 핵심 게임 로직

### 6.1 Snake 클래스

- **세그먼트 기반 이동:** 매 틱마다 머리 세그먼트를 앞에 추가, 꼬리 제거
- **부스트:** 길이 20 이상일 때 활성화, 속도 2배 + 세그먼트 2개씩 소모 (30% 확률로 먹이 드롭)
- **사망 조건:** 월드 경계 충돌 또는 다른 뱀 몸통 충돌
- **사망 시:** 세그먼트 3개마다 큰 먹이(big food) 드롭
- **스코어:** `세그먼트 수 - 초기 길이(20)`
- **바디 반경:** `min(6 + length * 0.08, 22)` — 길이에 비례하여 두꺼워짐

### 6.2 충돌 감지 (Spatial Grid)

- 월드를 200px 셀로 분할한 공간 해시 그리드 사용
- 매 틱마다 모든 뱀의 몸통 세그먼트를 그리드에 등록 (인덱스 8부터, 2개 간격)
- 머리-몸통 충돌 검사 시 인접 셀만 탐색하여 O(n) 수준으로 최적화
- 머리 근처 8개 세그먼트는 스킵 (자기 자신과의 충돌 방지)

### 6.3 AI 로직

- **먹이 탐색:** 300px 범위 내 가장 가까운 먹이를 향해 이동
- **위험 감지:** 120px 내에 자신보다 큰 뱀이 있으면 도주 + 부스트
- **경계 회피:** 맵 가장자리 400px 이내면 중앙 방향으로 회전
- **순환 감지:** 같은 먹이를 쫓으며 4π(2바퀴) 이상 회전하면 후퇴 모드
- **후퇴 모드:** 40~60틱 동안 반대 방향 이동 후 재탐색
- **스폰 위치:** 50% 확률로 플레이어 근처(800~2000px)에 생성

### 6.4 먹이 시스템

- 일반 먹이: 반경 5, 성장 +1
- 큰 먹이 (사망/부스트 드롭): 반경 9, 성장 +3
- 먹이 수 항상 800개 유지 (소비되면 자동 보충)
- 10가지 색상 랜덤 배정

## 7. 클라이언트 렌더링

### 7.1 카메라

- 플레이어 머리 위치를 부드럽게 추적 (보간 계수 0.15)
- 뱀 길이에 따라 줌아웃: `zoom = max(0.45, 1 - length * 0.0008)`

### 7.2 렌더링 최적화

- **shadowBlur 미사용:** 성능을 위해 그림자 효과 대신 반투명 원으로 글로우 표현
- **색상별 배치 렌더링:** 먹이를 색상별로 그룹화하여 fillStyle 변경 최소화
- **먹이 흔들림 애니메이션:** `sin/cos` 기반 위치 오프셋으로 생동감 표현
- **미니맵:** 별도 캔버스에 모든 뱀 위치 + 뷰포트 표시

### 7.3 뱀 렌더링

1. **글로우 레이어:** 반투명(20%) 큰 원으로 외곽 발광
2. **바디:** 2색 교대 패턴 (3개 세그먼트마다 색상 교체, 12가지 팔레트)
3. **머리:** 글로우 + 코어 원
4. **눈:** 흰자 + 검은 동공 (진행 방향 오프셋)
5. **이름:** 머리 위에 표시 (자신 또는 길이 25 이상)
6. **왕관:** 1위 뱀에게 금색 왕관 + 보석 장식

### 7.4 UI 요소

- **시작 화면:** 이름 입력, PLAY 버튼, 최고/최근 점수, 조작법 안내
- **인게임 HUD:** 점수/길이/킬, 리더보드 (상위 10), 미니맵, HOME 버튼
- **게임 오버:** 최종 점수, 최고 점수, PLAY AGAIN 버튼
- **#1 인디케이터:** 1위 뱀이 화면 밖에 있으면 화살표로 방향 표시 (가까우면 빨간색 깜빡임)

### 7.5 모바일 지원

- 터치 모드 2가지: **Touch** (화면 터치로 방향 제어), **Joystick** (가상 조이스틱)
- 부스트 버튼 (우하단)
- 반응형 CSS (768px 이하 모바일 레이아웃)
- 전체화면 자동 전환

## 10. 로컬 싱글플레이 버전 (`localGame.html`)

- 멀티플레이 버전과 동일한 게임 로직을 클라이언트 단독으로 실행
- 서버/WebSocket 불필요, 브라우저에서 바로 실행 가능
- `localStorage`에 최고 점수/최근 점수 저장
- 광고 슬롯은 플레이스홀더만 존재 (AdSense 스크립트 없음)
- 코드가 minified 스타일로 압축되어 있음

## 11. WebSocket 프로토콜

### 클라이언트 → 서버

| 메시지 | 필드 | 설명 |
|--------|------|------|
| `join` | `{type:"join", name}` | 게임 참가 |
| `input` | `{type:"input", a:angle, b:boost}` | 입력 (50ms 간격) |
| `respawn` | `{type:"respawn", name}` | 리스폰 |

### 서버 → 클라이언트

| 타입 | 필드 | 설명 |
|------|------|------|
| `w` (welcome) | `{t:"w", id, ws, p}` | 접속 성공 (ID, 월드크기, 팔레트) |
| `s` (state) | `{t:"s", s, f, y, lb, mm}` | 게임 상태 (뱀, 먹이, 내ID, 리더보드, 미니맵) |
| `d` (death) | `{t:"d", sc, l, k}` | 사망 (점수, 길이, 킬) |

### State 메시지 뱀 배열 포맷
```
s[i] = [id, name, segments(flat xy), paletteIdx, length, angle, boosting, score, kills]
```

### State 메시지 먹이 배열 포맷
```
f[i] = [x, y, color, radius]
```

## 13. 특징 및 완성도

### 잘 된 점
- Authoritative 서버 구조로 치팅 방지
- Spatial Grid 충돌 감지로 성능 최적화
- AI의 순환 감지/후퇴 등 자연스러운 행동
- shadowBlur 제거 등 렌더링 최적화
- 모바일 터치/조이스틱 지원
- 로컬 싱글플레이 버전 별도 제공
- Docker 배포 준비 완료
- 광고 슬롯 통합

### 개선 가능한 점
- 클라이언트 보간(Interpolation) 미구현 — 15Hz 전송으로 움직임이 끊길 수 있음
- 서버 단일 프로세스 — 대규모 동접 시 스케일링 한계
- HTTPS/WSS 미설정 — 프로덕션 배포 시 리버스 프록시 필요
- 방 시스템/매치메이킹 없음
- 사운드 효과 없음
- 로그인/계정 시스템 없음 (점수가 localStorage에만 저장)
